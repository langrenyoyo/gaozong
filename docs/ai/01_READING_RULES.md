# AI Project Reading Protocol

> 本文档是 AI Coding Agent 的项目阅读规范。
>
> 本规范优先级高于编码规范、测试规范和输出规范。
>
> 在完成阅读和理解前，禁止直接生成代码。

------

# 1. 核心原则

AI 的首要职责不是写代码。

AI 的首要职责是：

```text
理解项目
理解业务
理解调用链
理解影响面
```

然后才是：

```text
设计方案
实现代码
执行测试
```

如果理解错误：

```text
后续全部错误
```

因此：

```text
Reading First
Coding Later
```

------

# 2. 阅读目标

阅读的目标不是“看过文件”。

阅读完成后必须能够回答：

```text
1. 这个项目是什么？

2. 当前需求属于哪个模块？

3. 当前数据从哪里来？

4. 当前数据写到哪里去？

5. 当前业务流程是什么？

6. 当前权限在哪里校验？

7. 当前会影响哪些模块？

8. 当前风险等级是什么？

9. 最小修改方案是什么？

10. 如何验证修改正确？
```

如果无法回答以上问题：

```text
继续阅读
禁止编码
```

------

# 3. 阅读流程

收到任务后必须按顺序执行：

```text
理解需求
↓
项目级阅读
↓
模块级阅读
↓
调用链阅读
↓
业务语义阅读
↓
影响面分析
↓
输出阅读结果
↓
进入方案设计
↓
进入实现
```

禁止跳过阅读阶段直接编码。

------

# 4. 项目级阅读

适用于：

```text
首次进入项目
陌生项目
大型重构
跨模块需求
```

优先阅读：

```text
CLAUDE.md

docs/ai/05_PROJECT_CONTEXT.md

README.md

requirements.txt
package.json
pyproject.toml
```

如果涉及以下内容：

```text
部署
环境变量
Docker
数据库
权限
认证
```

额外阅读：

```text
docker-compose.yml

.env.development.example / .env.lan.example / .env.production.example

migration

认证模块

权限模块
```

输出：

```text
# 项目理解

项目类型：

核心业务：

主要用户：

核心模块：

技术栈：

部署方式：

高风险区域：
```

禁止无目的遍历整个项目。

禁止无目的阅读整个 docs 目录。

------

# 5. 模块级阅读

定位当前需求所属模块。

例如：

```text
用户管理

权限管理

订单系统

知识库

Agent

RAG

支付

文件管理

通知系统

评测系统
```

必须确认：

```text
当前模块负责什么？

输入是什么？

输出是什么？

依赖哪些模块？

被哪些模块调用？

是否涉及：

权限
数据库
配置
外部服务
缓存
消息队列
```

输出：

```text
# 模块理解

模块职责：

输入：

输出：

依赖：

被调用方：
```

------

# 6. 调用链阅读

必须找出真实调用链。

后端常见调用链：

```text
API
↓
Service
↓
Repository / CRUD
↓
Database
```

前端常见调用链：

```text
View
↓
Component
↓
Store
↓
API
↓
Backend
```

AI 应用常见调用链：

```text
User Input
↓
Agent / Router
↓
Retriever / Tool
↓
Prompt
↓
LLM
↓
Parser
↓
Response
```

必须确认：

```text
入口在哪里？

核心逻辑在哪里？

数据读取在哪里？

数据写入在哪里？

返回结果在哪里？
```

输出：

```text
# 调用链

用户操作：

入口文件：

核心逻辑：

数据读取：

数据写入：

返回结果：
```

调用链不清晰：

```text
禁止修改
继续阅读
```

------

# 7. 文件阅读规则

修改前至少阅读：

```text
1. 当前入口文件

2. 核心逻辑文件

3. 数据模型或类型定义

4. 同目录相邻实现

5. 测试文件

6. 配置文件（如果任务涉及配置）
```

禁止：

```text
只读一个文件就修改

只看函数名猜测逻辑

只根据目录结构推断业务
```

------

# 8. 相邻实现优先原则

新增功能前：

必须寻找同目录、同模块、同类型实现。

优先学习：

```text
命名风格

代码组织方式

异常处理方式

日志方式

权限校验方式

返回结构

测试风格
```

原则：

```text
一致性优先于优雅性
```

禁止重新发明一套风格。

------

# 9. 业务语义阅读

必须区分：

```text
技术对象

业务对象
```

例如：

```text
current_user

creator

owner

admin

email
```

不一定代表：

```text
业务负责人

任务归属人

通知接收人
```

必须确认：

```text
操作人是谁？

业务负责人是谁？

数据归属对象是谁？

权限校验对象是谁？

通知接收对象是谁？

是否存在代操作场景？
```

禁止：

```text
使用最容易获取的数据

替代业务上正确的数据
```

------

# 10. 高风险区域阅读

以下属于高风险：

```text
Docker

Docker Compose

Nginx

环境变量

数据库 Migration

认证

RBAC

支付

文件存储

缓存

异步任务

部署脚本

CI/CD
```

涉及高风险区域：

必须输出：

```text
# 高风险分析

当前实现：

修改原因：

影响范围：

风险等级：

验证方案：

回滚方案：
```

未完成风险分析：

```text
禁止修改
```

------

# 11. 数据库阅读规则

涉及数据库时：

必须阅读：

```text
Model

Schema

Repository / CRUD

Service

Migration

测试
```

必须确认：

```text
表结构

唯一索引

外键关系

历史数据

软删除逻辑

迁移方案

回滚方案
```

禁止：

```text
不了解表结构就修改数据库

不了解迁移历史就新增 Migration
```

------

# 12. 权限阅读规则

涉及权限时：

必须确认：

```text
有哪些角色？

权限在哪里校验？

前端是否仅隐藏按钮？

后端是否二次校验？

是否存在管理员代操作？

是否存在越权风险？
```

权限逻辑不清晰：

```text
禁止修改
```

------

# 13. AI / LLM 阅读规则

涉及：

```text
Prompt

Agent

RAG

Tool Calling

LLM
```

必须阅读：

```text
Prompt 模板

Prompt 版本

输入构造

检索逻辑

工具调用

输出解析

JSON Schema

Fallback 逻辑

测试样例
```

必须确认：

```text
输入是什么？

Prompt 如何构造？

输出是否结构化？

解析失败怎么办？

是否存在 Prompt Injection 风险？

是否需要稳定性测试？
```

------

# 14. 阅读结果输出

进入编码前必须输出：

```text
# 阅读结果

## 项目 / 模块理解

## 当前调用链

## 已阅读文件

## 业务语义确认

## 影响范围分析

## 风险等级

## 最小修改方案

## 验证方案
```

必须基于实际文件输出。

禁止：

```text
我已经理解项目

应该问题不大

看起来只需要小改
```

这类无依据结论。

------

# 15. 编码准入条件

只有同时满足以下条件：

```text
已完成阅读

已完成调用链分析

已完成业务语义确认

已完成影响面分析

已完成方案设计

已明确验证方案
```

才允许进入编码阶段。

否则：

```text
继续阅读
```

------

# 16. 最终原则

AI 必须先成为：

```text
项目读者
```

然后才能成为：

```text
项目开发者
```

在任何项目中：

```text
P0 阅读规范

P1 项目上下文

P2 执行规范

P3 测试规范

P4 输出规范
```

阅读规范拥有最高优先级。

# 17. P1-END-1 后额外阅读要求

P1-END-1 后，新窗口/新会话在完成标准阅读顺序后，**额外必须阅读**：

docs/ai/05_acceptance/P1_END_1_ACCEPTANCE.md

# 18. AI 文档自治维护规则

本节是 `docs/ai` 活动文档（CLAUDE.md、AGENTS.md、README.md、01~05 规则与上下文文件）的维护总规则。CLAUDE.md 与 AGENTS.md 中的"AI 文档自治维护要求"是本节的入口摘要；细则以本节为准。

## 18.1 维护责任

```text
AI 文档由 VibeCoding / AI Coding Agent 自主维护。

开发者不负责逐段人工修改文档，只负责：
  审阅 Git diff
  裁决高风险业务结论
```

发现文档与事实不符时，AI 必须主动修复，不得等待开发者指示。

## 18.2 先探索后更新

```text
禁止仅根据对话摘要、旧任务报告或现有汇总文档判断当前事实。

更新任何结论前，必须先定位对应的
代码 / 配置 / 迁移 / 测试 / 运行证据。
```

## 18.3 事实来源优先级

发生冲突时，从高到低：

```text
1. 已验证的实际运行证据
2. 当前有效代码
3. 数据库迁移和 schema
4. 当前配置、环境模板和 Compose
5. 自动化测试及验收脚本
6. 用户已经确认的业务决定
7. 当前专题文档
8. AI 汇总上下文
9. 历史报告和归档文件
```

## 18.4 每轮文档影响检查

每轮任务和每个大阶段完成后，必须执行一次文档影响检查：

```text
本轮改动是否使以下内容过期？

  05_PROJECT_CONTEXT.md 中的当前事实
  CLAUDE.md / AGENTS.md 中的硬约束
  README.md 导航
  相关专题文档
```

受影响 → 同轮更新；不受影响 → 明确说明"无文档影响"。

## 18.5 原位替换规则

```text
旧结论失效时：原位替换或删除。

禁止：
  在错误结论后追加"最新补充""当前更新"或相反结论
  保留旧错误结论并注明"以新内容为准"
  同一事实在活动上下文中保留多份结论
```

## 18.6 自主更新触发条件

出现以下任一情况必须更新文档：

```text
1. 服务、端口、部署入口、环境边界变化
2. 数据库、迁移、切换状态变化
3. 鉴权、权限、商户隔离边界变化
4. 发送门禁、安全 gate 口径变化
5. 阶段状态变化（完成 / 取消 / 延期 / 阻塞）
6. 用户确认了新的业务决定
7. 发现活动文档中存在事实错误或内部冲突
```

重复出现的问题和严重风险，应升级为长期规则写入对应规则文件。

## 18.7 不应更新的情况

```text
1. 未验证的计划、猜测、临时调试结论
2. 单次任务的详细过程（写任务报告，不写进上下文）
3. 提交号、测试通过数量、修改文件清单等流水账
   （仅在定位关键边界确有必要时保留提交号）
4. 与本轮任务无关、且未核验的他人结论
```

## 18.8 治理规则文件的较高修改门槛

```text
01~04 规则文件和本节规则属于治理层。

修改治理层需要满足：
  问题重复出现或风险等级高
  修改后不与现有规则冲突
  在任务报告中说明修改理由

日常事实更新只改 05_PROJECT_CONTEXT.md 和专题文档。
```

## 18.9 归档原则

```text
进入归档（docs/ai/archive/ 或专题目录）：
  已完成阶段的详细调查过程
  废弃架构和旧迁移方案
  历史运行证据和长测试报告
  仅用于解释"为什么曾经这样做"的内容

活动上下文最多保留一句仍然必要的历史提醒。

归档文件必须：
  头部注明"冻结快照 / 历史追溯用，非当前事实"
  可从 README.md 或专题入口找到
  不再维护"当前状态"
```

## 18.10 修改后一致性检查

每次修改活动文档后必须自查：

```text
1. 同一事实是否只剩一份结论
2. 是否引入了"已完成"与"进行中"并存
3. CLAUDE.md 与 AGENTS.md 硬约束是否语义一致
4. README.md 导航是否仍然有效
5. 是否误删了仍有效的生产、权限、数据、安全约束
```

## 18.11 任务报告要求

涉及文档维护的任务，报告中必须包含：

```text
修改了哪些文档、为什么
替换 / 删除了哪些旧结论
归档了什么
执行了哪些一致性检查
明确说明"无文档影响"的部分
```
