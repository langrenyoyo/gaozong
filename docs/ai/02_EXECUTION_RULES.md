# AI Coding Execution Rules

> 本文档是 AI Coding Agent 的执行规范。
>
> 在完成阅读规范后，所有代码修改必须遵守本规范。
>
> 本规范优先级低于：
>
> ```text
> 01_READING_RULES.md
> 05_PROJECT_CONTEXT.md
> ```
>
> 但高于测试规范和输出规范。

------

# 1. 核心原则

执行阶段目标：

```text
正确
稳定
可验证
可回滚
```

而不是：

```text
优雅
复杂
未来可能有用
```

原则：

```text
最小修改优先

一致性优先

可维护优先

可验证优先
```

------

# 1A. 阶段最终目标与边界总控

所有开发、测试、文档、前端、数据库、Local Agent 任务都必须先确认当前所属阶段，并遵守以下总控规则。

## 1A.1 阶段定义要求

1. 每个阶段必须有明确的最终目标。
2. 每个阶段必须说明“完成后系统应该达到什么状态”。
3. 每个阶段必须说明“不属于本阶段的事情”。
4. 每个阶段只能交付本阶段目标，不允许提前实现后续阶段功能。
5. 发现后续问题只能记录到风险 / 后续计划，不能擅自开发。

## 1A.2 阶段开始前必须复述

每个阶段开始前，必须先输出并确认：

```text
本阶段目标
本阶段允许修改范围
本阶段禁止事项
本阶段验收标准
是否需要用户确认
```

如果当前阶段需要用户确认，必须先等待确认，不得继续实现。

## 1A.3 阶段执行边界

禁止：

```text
把方案设计、迁移脚本、模型字段、业务状态机、导出、前端等多个阶段混在同一轮完成
用“顺便完成了某功能”替代阶段验收
因为发现后续问题而擅自扩大修改范围
在当前阶段提前实现后续阶段能力
```

如果发现当前阶段需要做后续阶段内容才能继续，必须停止并说明阻塞，不允许自行扩大范围。

## 1A.4 阶段结束后必须输出

每个阶段结束后，必须输出：

```text
是否达成本阶段最终目标
是否越界修改
是否提前实现后续阶段能力
修改文件列表
测试结果
风险点
下一阶段建议
```

------

# 2. 最小修改原则

只修改当前任务直接相关内容。

禁止：

```text
修复 Bug 时顺手重构

新增功能时顺手优化架构

修改接口时顺手调整命名

修复样式时顺手重写页面
```

要求：

```text
只修改必要文件

只修改必要逻辑

保持现有行为兼容
```

------

# 3. 禁止提前抽象

AI 最容易犯的错误：

```text
为未来设计

为可能出现的需求设计

为不存在的扩展设计
```

禁止：

```text
BaseXXX

AbstractXXX

PluginSystem

GenericPipeline

StrategyFactory

EventBus

Registry
```

除非已有多个真实使用场景。

------

## 抽象升级规则

```text
第 1 个使用者
直接实现

第 2 个使用者
提取公共函数

第 3 个使用者
考虑提取公共组件

第 4 个使用者
考虑抽象层
```

允许少量重复代码存在。

------

# 4. 默认禁止新增依赖

新增依赖前必须说明：

```text
为什么需要？

现有依赖为什么无法满足？

是否有标准库替代？

是否影响部署？

是否影响构建？

是否影响安全？
```

未完成说明：

```text
禁止新增依赖
```

------

# 5. 文件规模控制

建议：

```text
单文件 ≤ 500 行

单函数 ≤ 80 行
```

超过后考虑拆分。

拆分原则：

```text
按职责拆分

按模块拆分

按功能拆分
```

禁止：

```text
为了拆分而拆分

制造大量空目录

制造大量中间层
```

------

# 6. 保持现有架构

优先遵循项目已有结构。

例如：

```text
API
↓
Service
↓
Repository
↓
Database
```

或者：

```text
Controller
↓
Domain
↓
Persistence
```

禁止：

```text
新增架构层

替换已有架构模式

改变项目主要组织方式
```

除非任务明确要求。

------

# 7. 数据模型规则

原则：

```text
数据结构清晰

关系明确

避免动态结构
```

禁止：

```text
JSON 套 JSON

无约束动态字段

一个实体承担多个职责
```

新增持久化字段：

```text
必须遵循项目现有迁移机制
```

禁止绕过项目数据库管理方式。

------

# 8. 状态管理规则

所有状态变化必须可追踪。

禁止：

```text
任意修改状态字段

绕过状态流转逻辑

跳过业务校验直接改状态
```

要求：

```text
通过项目已有状态管理机制修改状态

保持状态流转可验证

保证状态变化可回溯
```

------

# 9. 异常处理规则

禁止：

```python
except:
    pass
```

禁止：

```python
except Exception:
    return "失败"
```

要求：

```text
记录日志

保留上下文

提供有意义错误信息

区分业务异常和系统异常
```

------

# 10. 外部服务调用规则

涉及：

```text
LLM

邮件

支付

第三方 API

搜索服务

对象存储
```

要求：

```text
超时控制

异常处理

降级方案

日志记录
```

禁止：

```text
直接将底层异常暴露给用户
```

# 11. Prompt 与 LLM 规则

涉及 AI 功能时：

要求：

```text
Prompt 版本化

Prompt 集中管理

输出结构化

结果可校验

失败可恢复
```

禁止：

```text
大量 Prompt 写死在代码中

依赖自由文本解析

无约束输出格式
```

------

# 12. 数据库变更规则

涉及数据库结构修改：

必须：

```text
使用项目既有迁移机制

提供升级方案

提供回滚方案

分析历史数据兼容性
```

禁止：

```text
直接修改生产结构

绕过迁移机制

破坏历史数据兼容性
```

------

# 13. 高风险变更规则

以下属于高风险：

```text
数据库

认证

权限

配置

环境变量

部署

缓存

异步任务

文件存储

支付

第三方服务
```

涉及以上内容：

必须先完成：

```text
现状分析

风险分析

修改方案

验证方案

回滚方案
```

未完成分析：

```text
禁止修改
```

------

# 14. 验证优先原则

任何修改必须具备验证方式。

至少回答：

```text
如何证明修改成功？

如何证明旧功能未损坏？

如何复现问题？

如何验证修复？
```

无法验证：

```text
禁止提交
```

------

# 15. 执行完成后必须输出

必须输出：

```text
# 修改内容

# 修改文件

# 修改原因

# 风险等级

# 影响范围

# 验证方案

# 回滚方案
```

禁止只输出：

```text
已经修改完成
```

------

# 16. 最终原则

AI 的职责不是创造新的系统。

AI 的职责是：

```text
理解已有系统

遵循已有系统

安全修改已有系统
```

任何时候：

```text
阅读规范
>
项目上下文
>
执行规范
>
测试规范
>
输出规范
```

优先级不得颠倒。

# 17. BUG 修复前置探索原则

修复 bug 之前，必须先探索代码，确认以下内容：

```text
1. 真实调用链：从入口到失败点的完整代码路径
2. 相关代码文件：涉及的所有文件和函数
3. 失败发生在哪一层：API / Service / UI Automation / OCR / 外部调用
4. 失败输入是什么：触发 bug 的具体参数、状态、环境
5. 失败输出是什么：错误信息、异常堆栈、错误状态码
6. 当前现象是否来自真实代码逻辑：
   - 是否来自缓存残留？
   - 是否来自旧版本 exe？
   - 是否来自旧版前端？
   - 是否来自旧进程未退出？
   - 是否来自旧配置未更新？
7. 是否需要人工辅助提供：
   - 日志文件
   - 截图
   - JSON 响应
   - 控制台输出
   - 接口响应
   - 复现实测结果
```

**在编写 bug 修复计划之前，必须先澄清以上全部内容。**

只有能说明**真实根因**或**高度可信的根因假设**时，才可以开始编写修复计划供用户审查。

禁止：

```text
仅凭现象描述就编写修复方案

未确认调用链就修改代码

未确认失败层级就跨层修复

假设旧代码一定有 bug 而不加验证
```

------

# 18. 修复计划审查前置条件

在修复计划执行之前，必须向用户澄清以下内容：

```text
1. 修改范围：
   - 涉及哪些文件
   - 涉及哪些函数
   - 涉及哪些配置

2. 是否涉及非相关代码文件：
   - 是否顺手重构了无关代码
   - 是否修改了本次 bug 以外的逻辑

3. 是否会引入新的 bug：
   - 修改是否影响其他调用方
   - 修改是否改变已有行为
   - 修改是否破坏边界条件

4. 是否可能导致回归：
   - 已有功能是否受影响
   - 已有测试是否需要更新

5. 需要新增哪些测试：
   - 回归测试
   - 边界测试
   - 异常路径测试

6. 需要保留哪些安全门禁：
   - 当前安全约束是否受影响
   - 是否需要新增安全检查

7. 是否需要人工验证：
   - 哪些步骤需要人工确认
   - 哪些结果需要人工检查
```

**修复计划必须经过用户审查后才能执行。**

# 19. 高风险代码日志原则

重要逻辑和风险大的代码逻辑**必须**强制写日志，以便后续 bug 排查和修复。

必须写日志的场景（包括但不限于）：

```text
1.  自动化操作微信窗口（打开、切换、关闭）
2.  前台焦点切换（置顶、前台切换、焦点丢失）
3.  OCR 模型加载（加载开始、成功、失败、耗时）
4.  OCR 识别结果（输入截图、输出文本、置信度、匹配结果）
5.  搜索框定位（定位策略、找到/未找到、坐标）
6.  搜索结果定位（搜索结果列表、第一项选择、结果验证）
7.  联系人验证（验证策略、verified/partial_match/unverified、OCR 原文）
8.  粘贴/发送前门禁（门禁检查结果、是否通过、拒绝原因）
9.  任务状态流转（状态变更、变更原因、变更时间）
10. 线索同步/派发（同步开始、同步结果、分配结果）
11. WechatTask 创建和结果回写
12. 任何 failure_stage 生成位置（失败阶段、失败原因、关键输入）
13. exe 启动入口（版本号、运行时路径、注册路由列表）
14. PyInstaller 打包后的运行时路径、版本、路由信息
15. 异常捕获和 finally 清理逻辑
```

日志要求：

```text
1. 不能只写"失败了"，必须写清楚：
   - stage（当前阶段）
   - 输入摘要（关键参数）
   - 关键判断结果（通过/未通过/拒绝原因）
   - failure_stage（失败阶段标识）

2. 不能记录敏感明文过多：
   - 手机号等敏感字段需要脱敏
   - 只保留必要诊断信息

3. 自动化相关日志必须能帮助判断：
   - 是否操作了正确机器
   - 是否操作了正确微信窗口
   - 是否通过联系人验证
   - 是否阻止了发送
   - 是否进入了安全门禁

4. 对于 exe 闪退、后台 Agent、跨机器调用：
   - 必须优先保证日志可见或可导出
   - 日志写入文件而非仅控制台输出
   - 异常时自动输出关键诊断信息
```

禁止：

```text
高风险逻辑不写日志

日志只写成功不写失败

日志只写"失败"不写原因

日志丢失关键判断结果

日志记录敏感明文
```

# 20. 微信自动化安全边界（当前有效）

以下安全边界适用于当前所有微信自动化任务，除非用户明确批准，不得放宽：

```text
1.  poll-and-execute 必须只处理 notify_sales
2.  poll-and-detect 必须只处理 detect_reply
3.  poll-and-detect 不调用 input_writer.py
4.  poll-and-detect 不写输入框、不按 Enter、不发送消息
5.  detect_reply（检测链路）必须保持只读：action.sent=false、action.pasted=false
6.  真实发送必须经后端 gate：联系人验证、前台焦点、违禁词替换、
    人工接管、限频、失败回写、幂等、紧急停止
    （一期已废止旧的"只允许 paste_only / sent=false"硬门禁）
7.  Aw3 唯一联系人限制仅约束 Local Agent 调试端点（run_local_wechat_test），
    业务发送不依赖 Aw3 硬编码，靠 gate 组合保护
8.  9000 不直接操作微信
9.  微信操作只发生在客户机本地 19000 Local Agent
10. poll-and-execute 与 poll-and-detect 共享运行锁，避免并发操作微信
```

# 21. task_id 指定执行规则

P1-AUTO-1D-FIX2/FIX3 引入的 task_id 机制是当前主线的核心执行方式。

```text
1. React 新建任务后必须将 createdTask.id 传给 Local Agent
2. poll-and-execute 收到 task_id 时按指定任务执行
3. poll-and-detect 收到 task_id 时按指定任务执行
4. 无 task_id 时 fallback 到 pending 队列拉取
5. 禁止在主链路中依赖旧 pending 队列头部
6. 旧 pending 数据不应直接清理，但主链路不应被其阻塞
```

------

# 22. 诊断接口安全序列化规则

P1-AUTO-1D-FIX4 引入的安全序列化机制。

```text
1. 诊断接口（search-debug、search-result-debug 等）不得返回原始 UIA 控件对象
2. 返回值必须经过 _safe_json_serialize 清洗
3. Exception 转为 {"type": ..., "message": ...}
4. UIA 控件只保留 name/class_name/control_type/bounding_rectangle
5. 循环引用转为 "<circular_ref>"
6. 诊断接口即使内部异常也应返回 200 + success=false，不应抛 500
7. numpy 标量/ndarray 必须转为 Python 原生类型
```
