# auto_wechat 产品化接入与开发计划

## 0. 项目定位

auto_wechat 当前已完成基本验证，下一阶段目标不是继续堆功能，而是将已验证能力接入主系统，形成可维护、可追踪、可演示、可逐步上线的产品化闭环。

本项目定位为：

> auto_wechat 是主系统的副系统，不重建主业务系统。
> 主系统负责业务正确性，Local Agent / exe 负责客户电脑本机微信执行。

核心原则：

1. 优先复用主系统已有代码、表结构、接口风格和依赖。
2. 尽量减少新增第三方库。
3. 先完成代码探索、能力边界确认和接口契约，再进入正式编码。
4. 所有高风险自动化操作必须有日志、状态回写和人工兜底。
5. Local Agent 只回传执行事实，不直接决定最终业务状态。
6. 主系统根据 Local Agent 回写结果更新线索、通知任务、回复检测等业务状态。
7. 每个开发阶段必须有明确目标、依赖、验收标准和不做事项，避免 VibeCoding 发散跑偏。

# 3. 真实需求 PRD 确认

## 3.1 PRD 确认目标

在正式编码前，需要基于主系统代码探索和 MVP 能力边界，确认真实需求 PRD。

不能只根据 demo 能力反推需求，也不能在需求不清楚的情况下直接扩展功能。

## 3.2 必须确认的问题

### 3.2.1 线索来源

需要确认：

- 线索来自主系统已有数据？回复：来自[[frontend](https://douyinapi.misanduo.com/)](https://douyinapi.misanduo.com/)
- 线索来自抖音 / 巨量广告 webhook？回复：巨量广告 webhook
- 线索需要主动拉取？回复：需要有主动拉取
- 线索需要被动接收？回复：自动拉取，拉取间隔暂定30S
- 是否存在多个数据源？回复：否
- 每个数据源是否有唯一线索 ID？回复：数据源未做入库，应考虑唯一ID

### 3.2.2 数据清洗与去重

需要确认：

- 是否需要清洗数据？回复：需要
- 是否需要过滤无效线索？回复：需要，避免无效线索
- 是否需要手机号去重？回复：需要，避免同一客户多个销售跟进
- 是否需要 open_id / external_lead_id 去重？回复：需要，避免同一客户多个销售跟进
- 同一线索重复进入时如何处理？回复：发送给原有跟进销售，若无则进入自动分配
- 同一客户多次留资如何处理？回复：更新资料，发送给原有跟进销售，若无则进入自动分配

### 3.2.3 销售分配规则

需要确认：

- 手动分配还是自动分配？回复：自动分配，分流避免同一销售接收过多线索无法及时处理
- 是否轮询分配？回复：按客户导入列表的顺序轮流分配
- 是否按门店分配？回复：暂不，但预留
- 是否按销售在线状态分配？回复：不，按列表轮流
- 是否按销售微信昵称分配？回复：不
- 销售微信昵称由谁维护？回复：由客户增删减改
- 分配失败如何处理？回复：分配失败自动重试，失败后再进入待人工分配

### 3.2.4 微信通知内容

需要确认：

- 发送给销售的线索模板是什么？
  回复：
  【新线索分配】
  客户：
  来源：
  内容：
  联系方式：
  备注：
  请尽快跟进客户，并在处理完成后回复确认消息。（此句应可配置）
- 是否需要支持模板配置？回复：暂不
- 是否需要包含手机号、姓名、来源、备注？回复：需要
- 是否允许自动发送？回复：允许，但应数据入库方便追溯和保存，这个数据应设置保存时间为半年，另外过期要避免雪崩
- 是否需要发送前人工确认？回复：否
- 备注：应可客户配置自动分配时间，避免夜间或非工作时间自动分配线索

### 3.2.5 回复检测规则

需要确认：

- 多久内回复算有效？回复：暂定30分钟内
- 什么内容算有效回复？回复：第一层关键字列表，第二层大模型兜底
- 初始关键词是否为“收到”“已添加微信”？回复：应可配置，但需要提醒客户不要与线索模版有关键字重复
- 是否需要关键词配置化？回复：需要
- 是否需要正则？回复：暂不考虑
- 是否需要 LLM 判断？回复：关键字未命中，需要大模型兜底
- 是否需要人工复核？回复：不需要，但需要数据入库方便追溯和保存，这个数据应设置保存时间为半年，另外过期要避免雪崩

### 3.2.6 超时策略

需要确认：

- 超时时间是多少？回复：暂定30分钟
- 超时后是否重新分配？回复：是
- 超时后是否回调数据源？回复：否，重新分配
- 超时后是否通知管理员？回复：应有一个表暂时给管理员
- 超时后是否允许人工标记已处理？回复：允许管理员进行人工处理

### 3.2.7 回调策略

需要确认：

- 是否需要对数据源进行回调？回复：需要
- 回调哪些状态？回复：待定，还未确认
- 回调失败是否重试？回复：重试5次，失败后标记人工处理
- 回调失败是否人工处理？回复：是
- 回调日志是否需要入库？回复：是，必须入库

### 3.2.8 前端展示能力

需要确认 React 最终需要展示：

- 线索列表

- 销售分配状态

- 发送结果
- 回复检测结果
- 超时列表

- 人工处理入口

- 日志查看入口

# 4. 产品形态与职责边界

## 4.1 最终产品形态

客户打开 React 前端查看线索、任务、状态和结果。

主系统 / auto_wechat 后端负责：

- 自动拉取或接收线索
- 线索清洗
- 线索去重
- 线索入库
- 分配线索
- 创建微信通知任务
- 向 Local Agent / exe 下发任务
- 接收客户端执行结果
- 创建或更新回复检测任务
- 更新线索状态
- 更新通知任务状态
- 更新回复检测状态
- 展示 React 状态
- 处理失败重试
- 处理人工兜底
- 必要时回调数据源

客户电脑 B 运行开发提供的 exe / Local Agent。

客户电脑 B 上的 exe / Local Agent 负责：

- 连接主系统
- 接收微信执行任务
- 检查本机微信状态
- 操作本机微信
- 搜索销售微信昵称
- 确认正确联系人
- 粘贴线索信息
- 发送线索信息
- 检测销售回复
- 判断本地执行结果
- 将执行事实回写主系统

## 4.2 职责边界

### 主系统负责

- 业务状态
- 线索状态
- 分配规则
- 超时规则
- 是否重分配
- 是否回调数据源
- 是否需要人工处理
- React 展示
- 报表统计

### Local Agent / exe 负责

- 本机微信可用性检测
- 本机微信 UI 自动化
- 执行任务
- 采集执行结果
- 采集失败原因
- 回写执行事实

### Local Agent / exe 不负责

- 不决定最终线索状态
- 不决定销售分配规则
- 不直接改业务数据
- 不直接回调外部数据源
- 不绕过主系统独立闭环
- 不在联系人未确认时发送消息
- 不在微信状态异常时继续操作

# 5. 主线流程

## 5.1 主系统主线流程

1. 主系统从数据源自动拉取或接收线索。
2. 主系统对线索进行清洗、去重、过滤。
3. 主系统将有效线索入库。
4. 主系统根据分配规则分配给销售。
5. 主系统创建微信通知任务。
6. 主系统将任务下发给客户本地电脑的 Local Agent / exe。
7. Local Agent / exe 执行微信通知任务。
8. Local Agent / exe 回写发送结果。
9. 主系统根据发送结果更新 lead_notifications 和 douyin_leads.status。
10. 主系统创建或更新回复检测任务。
11. Local Agent / exe 检测销售回复。
12. Local Agent / exe 回写回复检测结果。
13. 主系统根据结果更新 reply_checks 和 douyin_leads.status。
14. 如果有效回复，则线索进入跟进中。
15. 如果超时，则线索进入超时状态。
16. 如果需要，主系统进行回调反馈或重新分配。
17. React 前端展示完整状态。

## 5.2 Local Agent / exe 发送线索流程

1. Local Agent / exe 接收微信通知任务。
2. 检查本机微信是否可操作。
3. 检查微信窗口是否可见、非最小化、非隐藏。
4. 检查微信是否可前台操作。
5. 打开微信搜索框。
6. 输入销售微信昵称。
7. 确认搜索框焦点正确。
8. 搜索并确认销售联系人。
9. 打开对应聊天窗口。
10. 粘贴线索信息。
11. 发送线索信息。
12. 采集发送结果。
13. 回写主系统。

## 5.3 Local Agent / exe 回复检测流程

1. Local Agent / exe 接收回复检测任务。
2. 检查微信状态。
3. 定位对应销售聊天窗口。
4. 读取最近消息。
5. 判断是否存在销售有效回复。
6. 命中有效回复则回写 detected_valid。
7. 未命中但未超时则继续等待或下次检测。
8. 超时则回写 timeout。
9. 失败则回写 check_failed 或 manual_required。

# 10. Local Agent / exe 安全门禁

## 10.1 微信窗口门禁

如果出现以下情况，Local Agent / exe 必须停止业务自动化，并回写 manual_required：

- 微信窗口不可见
- 微信窗口最小化
- 微信窗口隐藏
- 微信窗口白屏
- 微信窗口无法置前
- 当前前台窗口不是微信
- 微信焦点丢失

## 10.2 搜索框门禁

如果出现以下情况，禁止粘贴和发送：

- 搜索框焦点未确认
- 疑似焦点落在聊天输入框
- 搜索框坐标不可信
- 搜索结果未确认
- 联系人未确认

## 10.3 发送门禁

只有满足以下条件才允许发送：

- 微信窗口可操作
- 搜索框焦点已确认
- 销售联系人已确认
- 聊天窗口已确认
- 输入框内容为本次任务线索信息
- 当前任务 ID 与输入内容匹配

## 10.4 失败回写要求

任何失败都必须回写：

- task_id
- failure_stage
- failure_reason
- manual_required
- 当前窗口状态
- 是否已粘贴
- 是否已发送
- 是否需要人工处理

# 11. 异常分支与处理策略

## 11.1 联系人找不到

状态：

- lead_notifications.status = contact_not_found
- douyin_leads.status = manual_required

处理方式：

- 前端展示失败原因
- 人工修改销售微信昵称
- 支持重新下发任务

## 11.2 微信不可操作

状态：

- lead_notifications.status = wechat_not_ready
- agent_tasks.status = manual_required

处理方式：

- 提示客户手动打开微信主窗口
- 不允许自动恢复隐藏或最小化微信后继续执行
- 人工确认后重新执行

## 11.3 搜索框焦点未确认

状态：

- lead_notifications.status = search_focus_not_verified
- agent_tasks.status = manual_required

处理方式：

- 不粘贴
- 不发送
- 回写诊断信息
- 前端提示人工处理

## 11.4 发送成功但未检测到回复

状态：

- douyin_leads.status = waiting_reply
- reply_checks.status = waiting

处理方式：

- 继续等待
- 到达超时时间后进入 timeout
- 根据规则重新分配或回调

## 11.5 Local Agent 离线

状态：

- agent_clients.status = offline
- agent_tasks.status = created 或 retrying

处理方式：

- 等待下一次心跳
- 前端展示 Local Agent 离线
- 支持人工处理

## 11.6 回调失败

状态：

- douyin_leads.status = callback_failed

处理方式：

- 记录回调日志
- 自动重试
- 超过次数后人工处理
