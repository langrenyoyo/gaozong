# 算力页面解耦与抖音自动回复用量优化设计

## 1. 目标

本设计覆盖四个独立但有先后依赖的阶段：

1. 用模型供应商返回的真实 Token 作为算力点数计量基础。
2. 将商户算力流水收敛为面向业务的独立展示合同。
3. 接通管理员算力配置的路由、导航、权限和现有页面。
4. 在每条自动回复仍真实调用模型的前提下，降低重复提示和重复重试造成的输入 Token。

不使用固定模板绕过模型，不降低客户记忆、安全后处理、人工接管、频率、幂等和真实发送校验。

**实施状态：** 阶段一“真实 Token 计量”与阶段二“商户流水解耦”已完成；阶段三、阶段四仍按本设计顺序待实施。阶段一已落地 PostgreSQL `0014`、SQLite `0033`、旧请求兼容、供应商用量优先、估算标记及抖音重试阶段独立入账。阶段二已落地商户公开 7 字段流水投影（`list_merchant_transactions`）、9000 与独立算力服务商户路由收敛、前端五列表格与字段白名单静态门禁，内部计量与诊断字段不再向普通商户公开。

## 2. 当前事实

### 2.1 算力账本

- 9000 的 `compute_accounts` 和 `compute_transactions` 是算力账户与流水真源。
- 商户接口已经从可信鉴权上下文读取 `merchant_id`，不存在由前端指定商户读取流水的问题。
- 商户流水响应只返回业务化 7 字段投影（类型、中文使用场景、算力点数变动、变动后余额、时间等），模型、能力编码、原始字符数、智能体编号、会话编号和内部备注仅保留在内部账本，展示合同已与内部计费合同解耦。
- `actual_tokens` 已收敛为上浮前基础用量；新流水由 `usage_measurement_method` 区分供应商 Token、估算 Token 和历史字符量。

### 2.2 管理员算力配置

- 套餐管理、商户充值、套餐发放和六类计费上浮比例的后端接口已经存在。
- `SuperComputeConfig` 页面已经实现上述主要操作。
- 真实管理员只显示 `SideNav.adminItems`，但其中没有算力配置。
- `App.tsx` 的管理员路由、默认落点和 `Index.tsx` 的管理员内容分发也没有算力配置，因此页面对真实管理员不可达。
- 套餐、充值和发放目前只允许超级管理员；计费比例允许超级管理员或 `auto_wechat:admin:compute_config`。

### 2.3 抖音自动回复用量

- 每次模型调用都会发送系统提示、智能体提示、动态客户上下文和结构化输出要求。
- 智能体提示词约 1900 字时，即使只有约 200 字历史，当前请求也可达到约 5300 字符。
- 历史消息最多 10 条、总计最多 2500 字符，因此单次记录达到 7000～8000 字符符合当前实现。
- 历史信息纠正和手机号引导可能分别触发重试；每次重试都再次携带完整首轮消息并独立计费。
- 模型客户端透传的 `usage` 已进入计费上报；仅在供应商没有有效用量时回退估算。

## 3. 总体方案

采用“先校准计量，再改展示，最后优化提示”的顺序。没有真实 Token 基线时，不以字符流水下降替代真实优化结论。

### 3.1 调用边界

```text
抖音客户消息
  -> 9000 构造可信智能体、历史和客户记忆
  -> 9100 调用模型
  -> 9100 解析供应商 usage 并上报单次调用用量
  -> 9000 写入商户算力流水
  -> 商户接口返回业务化流水投影
  -> 管理员接口返回配置与必要的内部诊断信息
```

9000 继续负责商户隔离和计费入账；9100 不直写 9000 数据库；前端不持有内部令牌、不直连 9100 或独立算力服务。

## 4. 阶段一：真实 Token 计量

### 4.1 用量解析

每次成功模型调用按以下优先级确定基础用量：

1. 使用供应商 `usage.total_tokens`。
2. 若缺少总量，但输入与输出 Token 均有效，使用两者之和。
3. 若供应商未返回有效用量，使用字符估算，并标记为估算值。

兼容常见字段别名，但只接受非负整数；异常结构不得影响回复主链路，必须记录不含原文的诊断日志。

字符估算不引入模型专用分词依赖。中文字符按一个估算 Token 计算，连续英文、数字和结构化字段按每四个字符约一个估算 Token 计算，结果至少为 1。估算只作为供应商缺失用量时的兜底，并明确标记，不能显示为真实 Token。

### 4.2 流水字段

保留现有 `actual_tokens` 列以兼容历史数据，但将其定义收敛为“计费前基础用量”，由新字段说明单位和来源。新增可空字段：

- `usage_measurement_method`：`provider_tokens`、`estimated_tokens`、`legacy_characters`。
- `prompt_tokens`：供应商返回的输入 Token。
- `completion_tokens`：供应商返回的输出 Token。
- `cached_tokens`：供应商返回的缓存命中 Token，缺失时为空。
- `llm_call_stage`：`primary`、`retry_known_customer`、`retry_phone_goal`、`retry_combined`。

迁移时，仅把历史 AI 消耗流水的 `usage_measurement_method` 回填为 `legacy_characters`；不伪造历史输入、输出或缓存 Token。

`delta_tokens` 继续表示应用上浮比例后的算力点数变动。每次真实模型调用独立入账，重试不会与首次调用合并收费。

### 4.3 内部上报合同

`/internal/compute/usage` 保持旧调用兼容，新增可选用量明细字段。旧客户端只传 `tokens` 时仍能工作；新 9100 客户端同时传计量方式、输入、输出、缓存和调用阶段。

### 4.4 验收

- 有 `usage.total_tokens` 时，基础用量等于供应商总 Token。
- 只有输入和输出 Token 时，基础用量等于两者之和。
- 无 `usage` 时走估算并标记 `estimated_tokens`。
- 同一客户消息发生一次重试时产生两条流水，阶段分别可识别。
- 上报失败不阻断 AI 回复，不改变发送决策。

## 5. 阶段二：商户流水解耦

### 5.1 商户响应合同

商户流水接口返回独立投影，只保留：

- `id`
- `type`：稳定类型编码
- `type_label`：充值、套餐发放、消耗
- `business_scene`：例如抖音自动回复、知识问答、微信助手
- `points_change`
- `balance_after`
- `created_at`

不返回模型、能力编码、基础用量、计量方式、智能体编号、会话编号和内部备注。未知业务来源统一显示“AI 服务”，不得直接回显未知内部编码。

### 5.2 商户页面

页面列固定为：类型、使用场景、算力点数变动、变动后余额、时间。继续支持分页、刷新、加载、空数据和错误状态。

商户不承担内部计量诊断职责。真实 Token、重试阶段和模型信息只保留在内部或管理员诊断范围。

### 5.3 验收

- 商户 A 无法读取商户 B 的流水。
- 商户响应和页面中不出现模型名称、内部能力编码、字符数、智能体编号或内部备注。
- 充值、套餐发放和 AI 消耗均能得到稳定中文业务场景。

### 6.1 路由与导航

新增单一管理员入口 `/admin/compute-config`，管理员导航显示“算力配置”。页面内部使用三个切换视图，不新增侧边二级导航：

- 计费比例
- 套餐管理
- 商户发放

原 `/compute/packages` 和 `/compute/markup-ratios` 可保留兼容跳转，但不再作为管理员主入口。

### 6.2 权限

以下操作统一允许超级管理员或精确权限 `auto_wechat:admin:compute_config`：

- 查看和修改计费比例
- 查看、创建和修改套餐
- 给商户充值算力点数
- 给商户发放套餐

普通商户权限和其他管理员权限均不得访问。所有写操作记录操作者、操作类型、目标摘要和失败阶段，不记录内部令牌。

### 6.3 页面复用

复用现有 `SuperComputeConfig`、算力接口封装和表单逻辑，仅调整入口、内部切换和权限分发。商户标识统一称为“商户编号”，不展示代码变量名。

### 6.4 验收

- 仅有算力配置权限的管理员登录后默认可进入该页面。
- 管理员导航存在“算力配置”。
- 三个视图均能完成现有操作。
- 无权限账号直接访问路由和调用接口均返回拒绝。

## 7. 阶段四：提示词与重试优化

### 7.1 约束

- 每条需要自动回复的客户消息仍真实调用模型。
- 不使用固定模板代替模型。
- 不删除智能体业务目标、客户记忆、安全规则或结构化输出校验。
- 不因优化改变真实发送门禁。

### 7.2 提示压缩

将提示拆成稳定系统前缀和动态用户上下文：

- 智能体提示只注入一次。
- 合并全局规则与绑定智能体分支中重复的安全、记忆和输出要求。
- 删除传给模型但不参与回答的 tenant、merchant、account、agent 等内部编号。
- `latest_customer_message` 与 `customer_message` 只保留一个。
- 将 `known_customer_requirements`、`known_customer_info` 和 `must_not_ask_again` 合并为紧凑客户上下文，避免同一信息重复出现。
- 结构化输出字段只声明一次。
- 历史策略只在系统提示中声明一次。

稳定系统前缀始终放在消息首部，为供应商自动提示缓存创造条件，但阶段四不依赖供应商缓存才能成立。

### 7.3 历史与记忆

客户长期信息继续由结构化记忆提供；近期历史用于承接语气、问题和最近回复。初始目标为最近 6 条、最多 1200 字符，并通过回归测试确认不会丢失预算、车型、年份、城市、联系方式和最近追问。

若测试证明 6 条不足，优先提高窗口，而不是删除结构化记忆或放宽安全规则。

### 7.4 重试合并

首次回复完成后一次性检查：

- 是否重复询问已知客户信息。
- 是否遗漏当前智能体明确要求的手机号引导。

任一失败时构造一个合并纠正请求，每条客户消息最多追加一次模型重试。重试仍携带必要系统规则和紧凑客户上下文，但不重复无关内部字段。

### 7.5 验收

- 每条正常自动回复至少调用一次模型，绝不走固定模板。
- 无纠正需要时只产生一次模型调用和一条用量流水。
- 同时命中两种纠正条件时最多追加一次重试。
- 平均输入 Token 相比阶段一基线降低至少 30%。
- 客户记忆、手机号目标、提示注入防护、高风险内容阻断和真实发送门禁回归通过。

## 8. 测试策略

- 算力服务：计量解析、估算、上浮、历史兼容、商户隔离和权限测试。
- 9100：供应商用量字段兼容、每次调用阶段、合并重试和提示长度断言。
- 前端：商户流水字段不可见、管理员路由可达、无权限拒绝、三视图切换。
- 集成验证：同一条简单消息和一条需要纠正的消息分别核对模型调用次数、真实 Token、流水条数和最终发送结果。

## 9. 风险与回滚

- 数据库迁移属于高风险变更，必须先在开发 PostgreSQL 执行 Alembic 升级与回滚演练；生产不得使用 `create_all`。
- 供应商 `usage` 结构可能不同，解析失败时只能回退估算，不能阻断回复。
- 提示压缩可能影响回复质量，阶段四必须保留阶段一基线并支持整阶段回滚到旧提示构造。
- 管理员权限扩大仅限精确算力配置权限，不得用“任意管理员权限”代替。
